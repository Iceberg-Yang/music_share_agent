# 双人音乐抽签 Agent MVP 产品规划与设计规范

## 1. 项目一句话

一个面向两个人的 AI 音乐抽签分享应用：用户创建可分享房间后，系统根据默认主题、氛围描述或聊天记录生成发散主题，两位参与者分别抽取主题并分享一首相关音乐，最终生成可分享的双人音乐结果页与 AI 氛围总结。

## 2. 产品背景

音乐分享天然适合轻社交，但普通歌单分享缺少互动感和话题感。本项目用“抽签”降低参与门槛，用“主题”制造表达空间，用“AI Agent”把聊天记录、情绪、关键词转化为更个性化的音乐分享场景。

MVP 不追求复杂社交网络，而是聚焦一个完整闭环：

- 两个人。
- 一个分享链接。
- 一次抽签。
- 两首歌。
- 一个可分享结果页。

## 3. 产品目标

### 3.1 核心目标

- 支持用户创建双人音乐抽签房间。
- 支持通过链接邀请第二位用户加入。
- 支持默认发散主题池，例如天空、颜色、地区、心情、开车、公路、雨天等。
- 支持预留聊天记录分析能力，后续可接入微信聊天记录文本。
- 支持大模型根据聊天记录或氛围描述生成主题。
- 支持两位用户各自抽取一个不重复主题。
- 支持用户提交歌曲、歌手、链接和推荐理由。
- 支持生成结果页和 AI 双人音乐氛围总结。
- 支持预留网易云音乐歌单接入接口。

### 3.2 非目标

MVP 阶段暂不实现：

- 多人房间。
- 实时聊天室。
- 复杂好友关系。
- 微信自动读取聊天数据库。
- 强依赖网易云音乐自动加入歌单。
- 完整音乐版权播放能力。
- 复杂 Agent 自主规划。

## 4. 目标用户

### 4.1 主要用户

- 想和朋友、伴侣、同学轻松分享音乐的人。
- 想把聊天氛围转化成音乐主题的人。
- 想做一个可展示 AI Agent 能力 Demo 的开发者。

### 4.2 使用场景

- 两个朋友晚上聊天后，各自抽一个主题分享歌。
- 情侣根据当天聊天记录生成音乐主题。
- 面试或作品集展示中，演示 LLM 如何参与产品流程。
- 社交媒体分享一个“双人音乐结果页”。

## 5. MVP 用户故事

### 5.1 创建者

作为创建者，我希望创建一个音乐抽签房间，并把链接发给另一个人，让我们可以各自抽主题、分享音乐。

验收标准：

- 可以输入房间名称和昵称。
- 可以选择默认主题或输入氛围描述。
- 可以可选粘贴聊天记录。
- 创建成功后获得分享链接。

### 5.2 参与者

作为参与者，我希望通过链接加入房间，抽到一个主题，并提交一首相关歌曲。

验收标准：

- 可以输入昵称加入房间。
- 房间最多只能有两个人。
- 每个人只能抽一次。
- 两个人抽到的主题不重复。

### 5.3 结果查看者

作为结果查看者，我希望看到两个人分别抽到了什么主题、推荐了什么歌，以及 AI 对这场音乐局的总结。

验收标准：

- 结果页展示两位参与者的主题、歌曲、歌手、链接和推荐理由。
- 两人都提交后可生成 AI 总结。
- 结果页可通过链接分享。

## 6. 核心玩法

1. 创建者创建房间。
2. 系统准备主题池。
3. 创建者分享房间链接。
4. 第二位用户加入。
5. 两位用户分别点击抽签。
6. 每人获得一个发散主题。
7. 每人提交一首与主题相关的音乐。
8. 系统生成结果页。
9. 可选调用大模型生成双人音乐总结。

## 7. 主题设计规范

### 7.1 主题风格

主题应该短、开放、发散，适合引发个人联想，而不是像命题作文一样过度限定。

推荐类型：

- 天气：雨天、风、晴天、雾、暴雨。
- 颜色：蓝色、红色、白色、黑色、霓虹。
- 空间：天空、海边、房间、便利店、车站。
- 地区：北方、南方、东京、海岛、山里。
- 时间：凌晨、清晨、夏天、冬天、周末。
- 动作：开车、等待、离开、散步、逃跑。
- 心情：自由、孤独、热烈、松弛、失眠。
- 意象：旧照片、月亮、公路、火车、宇宙。

### 7.2 默认主题池

MVP 可内置以下主题：

```json
  "雨天",
  "海边",

  "开车",

  "夜晚",
  "清晨",
  "宇宙",

  "离开",
  "等待",
  "自由",

  "热烈",

  "旧照片",
  "便利店",

  "风",
  "霓虹灯",

  "日落",

```

### 7.3 AI 生成主题约束

大模型生成主题时需要遵守：

- 每个主题 2-5 个中文字为主。
- 优先生成名词、地点、颜色、天气、动作或状态。
- 不生成过于具体的问题，例如“分享一首适合失恋时听的歌”。
- 不生成敏感、攻击性或隐私暴露主题。
- 尽量避免重复和同义堆叠。
- 一次生成 12-20 个主题。

## 8. 页面规划

### 8.1 首页 / 创建房间页

页面目标：让用户快速创建一个双人音乐抽签房间。

核心组件：

- 产品标题。
- 简短说明。
- 房间名称输入框。
- 创建者昵称输入框。
- 氛围描述输入框，可选。
- 聊天记录粘贴框，可选。
- 主题生成方式选择：
  - 使用默认主题。
  - 根据氛围描述生成。
  - 根据聊天记录生成。
- 创建按钮。

关键状态：

- 空表单。
- 生成主题中。
- 创建成功。
- 创建失败。

### 8.2 房间等待页

页面目标：展示房间状态，并引导创建者分享链接。

核心组件：

- 房间名称。
- 当前参与人数。
- 分享链接复制按钮。
- 参与者列表。
- 等待第二人加入提示。

关键状态：

- 只有一人，等待加入。
- 两人已加入，可以开始抽签。
- 房间已满。

状态同步策略：

MVP 不引入 WebSocket，使用前端定时轮询获取房间状态：

- 轮询接口：`GET /api/rooms/:inviteCode`
- 轮询间隔：每 3 秒一次。
- 触发条件：页面处于等待状态时开启轮询，检测到另一人已加入或对方已提交音乐后停止轮询并更新页面。
- 页面不可见时（切换 Tab）暂停轮询，可见后恢复，使用 `document.visibilitychange` 监听。
- 轮询失败时静默重试，不向用户展示网络错误，避免干扰体验。

### 8.3 加入房间页

页面目标：让第二位用户通过链接加入房间。

核心组件：

- 房间名称。
- 昵称输入框。
- 加入按钮。
- 房间人数提示。

异常状态：

- 房间不存在。
- 房间已满。
- 昵称为空。
- 昵称重复。

### 8.4 抽签页

页面目标：让用户抽取一个主题。

核心组件：

- 当前用户昵称。
- 抽签按钮。
- 抽签动效。
- 抽到的主题卡片。
- 去提交音乐按钮。

规则：

- 每人只能抽一次。
- 两人不能抽到同一主题。
- 抽中后结果写入后端，不依赖前端状态。

### 8.5 提交音乐页

页面目标：收集用户与主题相关的音乐选择。

核心组件：

- 已抽到的主题。
- 歌名输入框。
- 歌手输入框。
- 音乐链接输入框，可选。
- 推荐理由输入框，可选。
- 提交按钮。

表单校验：

- 歌名必填。
- 歌手建议必填。
- 链接可为空。
- 推荐理由可为空。

### 8.6 结果页

页面目标：展示最终双人音乐结果，并形成可分享内容。

核心组件：

- 房间名称。
- 两张结果卡片。
- 每张卡片展示昵称、主题、歌名、歌手、链接、推荐理由。
- AI 氛围总结。
- 关键词标签。
- 复制分享链接按钮。
- 导出歌单文本按钮。

结果页状态：

- 等待另一人提交。
- 两人已提交，展示完整结果。
- AI 总结生成中。
- AI 总结生成失败，可重试。

## 9. Agent 设计

MVP 中的 Agent 应该被设计为“受控工作流”，而不是完全自主行动的复杂 Agent。核心价值是将大模型放在最能体现语义理解和生成能力的环节。

### 9.1 Agent 模块划分

#### Chat Analyzer

职责：

- 分析用户粘贴的聊天记录。
- 提取情绪、关键词和关系氛围。
- 为主题生成提供结构化上下文。

输入：

```json
{
  "chatText": "聊天记录文本",
  "participants": ["A", "B"],
  "roomMoodHint": "想要轻松一点"
}
```

输出：

```json
{
  "mood": "轻松、怀旧、带一点调侃",
  "keywords": ["旅行", "天气", "工作", "夜晚"],
  "relationshipTone": "熟悉、自然、有互相调侃",
  "topicDirection": ["公路", "城市", "夜晚", "松弛"]
}
```

#### Topic Generator

职责：

- 根据默认配置、氛围描述或聊天分析结果生成主题池。
- 输出可用于抽签的主题数组。

输入：

```json
{
  "mood": "轻松、怀旧",
  "keywords": ["旅行", "天气", "工作"],
  "topicDirection": ["公路", "城市", "夜晚"],
  "count": 16
}
```

输出：

```json
{
  "topics": ["公路", "蓝色", "雨天", "便利店", "凌晨", "南方", "旧照片", "风"]
}
```

**LLM 调用合并优化：**

Chat Analyzer 和 Topic Generator 可以合并为一次 LLM 调用，减少延迟和 Token 消耗：

- 当聊天记录较短时（少于 200 条），直接一次调用同时输出 `mood + keywords + topics`。
- 当聊天记录很长（超过 200 条）时，先用低成本模型做摘要，再把摘要喂给主模型生成主题。
- MVP 阶段可以直接用合并调用，进阶版再考虑拆分。

合并后的统一输出结构：

```json
{
  "mood": "轻松、怀旧、带一点调侃",
  "keywords": ["旅行", "天气", "工作", "夜晚"],
  "relationshipTone": "熟悉、自然、有互相调侃",
  "topics": ["公路", "蓝色", "雨天", "便利店", "凌晨", "南方", "旧照片", "风"]
}
```

#### Draw Host

职责：

- 控制抽签规则。
- 防止重复抽取。
- 记录抽签结果。
- 判断房间流程状态。

说明：

这部分不需要大模型，应该由确定性业务逻辑实现。

#### Summary Writer

职责：

- 根据两位用户的主题、歌曲和推荐理由生成结果总结。
- 输出适合结果页展示和分享的短文案。

输入：

```json
{
  "roomName": "今晚听什么",
  "entries": [
    {
      "nickname": "A",
      "topic": "公路",
      "songName": "一路向北",
      "artist": "周杰伦",
      "reason": "很适合晚上开车听"
    },
    {
      "nickname": "B",
      "topic": "蓝色",
      "songName": "Blue",
      "artist": "Yung Kai",
      "reason": "有一种安静的感觉"
    }
  ]
}
```

输出：

```json
{
  "summary": "这场音乐局有一种从夜路开向安静海边的感觉。一首偏向移动和远方，另一首更像停下来之后的情绪。",
  "tags": ["夜晚", "公路", "安静", "松弛"]
}
```

**Summary Prompt 设计原则：**

总结文案是用户最终会看到并分享的内容，直接决定产品质感，Prompt 需要明确约束生成风格：

- **字数控制**：总结控制在 50-80 字之间，不超过两句话。太长会稀释氛围感，太短缺乏诗意。
- **视角**：用第三人称或旁观者视角描述这场音乐局，不要直接对用户说"你们"。
- **意象优先**：鼓励使用场景感、画面感的表达，例如"夜路、海边、窗帘、月台"，而不是抽象评价。
- **禁止套话**：不输出"你们真的很有默契""这两首歌都很好听""非常适合"之类的空洞文案。
- **不直接评价音乐**：不说歌好不好，不说歌手怎么样，只描述两首歌组合在一起呈现的氛围。
- **保留模糊感**：不过度解释为什么，留一些空白给读者自己感受。
- **Tags 要精简**：输出 3-5 个标签，每个 2-4 个汉字，类似"夜晚、安静、松弛"，不输出长短语。

### 9.2 是否需要 LangChain / LangGraph

MVP 阶段不建议一开始引入 LangChain。

原因：

- 当前流程较短，主要是聊天分析、主题生成、结果总结三类模型调用。
- 抽签、房间状态、提交歌曲都是确定性业务逻辑。
- 直接封装 `LLMService` 更容易控制 Prompt、结构化输出、错误处理和部署。
- 对简历项目来说，清晰的产品闭环比堆框架更重要。

推荐 MVP 架构：

```text
Frontend
  -> API Routes / Backend Service
    -> Room Service
    -> Draw Service
    -> LLM Service
      -> analyzeChat()
      -> generateTopics()
      -> summarizeResult()
```

适合后续引入 LangGraph 的时机：

- 需要接入多个工具，例如聊天分析、音乐搜索、网易云歌单写入、分享卡片生成。
- 需要状态机式流程管理。
- 需要可观测的 Agent 执行链路。
- 需要对 Agent 节点做重试、回滚、条件分支。
- 需要把项目升级成更完整的 Agent Workflow 展示。

进阶版 LangGraph 流程可设计为：

```text
Create Room
  -> Analyze Chat
  -> Generate Topics
  -> Wait For Draw
  -> Collect Music Entries
  -> Match Music Platform Tracks
  -> Add To Playlist
  -> Generate Summary
  -> Publish Result
```

## 10. 微信聊天记录接口预留

### 10.1 MVP 处理方式

MVP 不直接读取微信本地数据库。推荐先支持用户手动粘贴聊天记录或上传导出的文本文件。

原因：

- 微信本地记录读取涉及加密数据库和平台差异。
- 直接读取聊天记录存在明显隐私和授权问题。
- 粘贴文本足够展示 LLM 对聊天语境的分析能力。

### 10.2 接口草案

```http
POST /api/chat/analyze
```

请求体：

```json
{
  "roomId": "room_123",
  "source": "wechat_text",
  "content": "聊天记录文本",
  "participants": ["A", "B"]
}
```

响应体：

```json
{
  "mood": "轻松、怀旧、带一点调侃",
  "keywords": ["旅行", "天气", "工作", "旧朋友"],
  "topics": ["蓝色", "公路", "南方", "雨天", "便利店", "凌晨"],
  "reason": "聊天中多次出现出行、天气和疲惫感，因此生成偏流动、城市和夜晚的音乐主题。"
}
```

### 10.3 隐私提示

产品界面需要明确提示：

- 聊天记录仅用于生成主题。
- 建议用户删除手机号、地址、账号等敏感信息。
- MVP 可默认不长期保存原始聊天记录，只保存分析结果和生成主题。

## 11. 网易云音乐歌单接口预留

### 11.1 MVP 处理方式

MVP 不强依赖网易云音乐自动加入歌单，优先提供稳定的歌单导出能力。

结果页提供：

- 复制歌单文本。
- 导出 JSON。
- 导出 CSV。
- 展示待添加歌单列表。

### 11.2 扩展接口草案

```http
POST /api/music/netease/add-to-playlist
```

请求体：

```json
{
  "roomId": "room_123",
  "playlistId": "123456",
  "tracks": [
    {
      "songName": "一路向北",
      "artist": "周杰伦"
    }
  ]
}
```

预期流程：

1. 根据歌名和歌手搜索网易云音乐候选歌曲。
2. 返回候选结果给用户确认。
3. 用户选择正确歌曲。
4. 系统调用歌单接口加入指定歌单。
5. 返回成功列表和失败列表。

### 11.3 风险说明

网易云音乐公开 API 能力有限，第三方接口可能存在稳定性、登录态和合规风险。因此该能力适合放在扩展阶段，不作为 MVP 阻塞功能。

## 12. 信息架构

### 12.1 Room

```ts
type Room = {
  id: string
  name: string
  inviteCode: string
  status: "waiting" | "ready" | "drawing" | "submitted" | "completed"
  topicSource: "default" | "ai"
  topics: string[]
  chatAnalysis?: {
    mood: string
    keywords: string[]
    relationshipTone?: string
  }
  aiSummary?: string
  aiTags?: string[]
  createdAt: string
  updatedAt: string
  expiresAt: string
}
```

房间状态说明：

| 状态 | 含义 | 触发条件 |
|---|---|---|
| `waiting` | 只有创建者一人，等待第二人加入 | 房间创建后默认 |
| `ready` | 两人都已加入，可以开始抽签 | 第二人加入成功后 |
| `drawing` | 至少一人已完成抽签 | 任意一人完成 draw 后 |
| `submitted` | 两人都已提交音乐，等待生成总结 | 两条 MusicEntry 都存在时 |
| `completed` | AI 总结生成完毕，房间结束 | summary 生成成功后 |

房间过期设计：

- 房间创建时记录 `expiresAt`，默认设为创建时间 + 24 小时。
- 超过 `expiresAt` 且状态不为 `completed` 的房间，定时任务或查询时标记为过期归档。
- 过期房间不删除数据，只标记状态，保留结果页访问能力。
- 推荐在 Prisma 中加 `status: "expired"` 枚举值，或用单独 `isExpired` 字段。

### 13.4 抽签

```http
POST /api/rooms/:roomId/draw
```

请求体：

```json
{
  "participantId": "participant_1"
}
```

返回：

```json
{
  "topic": "公路"
}
```

并发抽签处理：

两个用户同时点击抽签时，可能抽到同一个主题。需要在数据库层保证原子性：

- 抽签逻辑使用数据库事务（`$transaction`）执行：
  1. 查询当前房间已被抽走的主题列表。
  2. 从剩余主题中随机选一个。
  3. 将该主题写入当前 Participant 的 `drawnTopic`。
  4. 以上三步在同一事务内完成，防止并发读写冲突。
- 若事务失败（例如主题已被另一人抢占），返回错误并提示用户重新抽签。
- 使用 SQLite 时，事务已天然串行；使用 PostgreSQL 时，建议在查询时加 `FOR UPDATE` 行锁。

这是面试中经常被追问的并发问题，建议在代码注释中保留对应说明。

## 14. 产品规则

- 一个房间最多两位参与者。
- 创建者自动成为第一位参与者。
- 第二位参与者通过链接加入。
- 昵称不能为空。
- 同一房间内昵称建议不重复。
- 每位参与者只能抽一次。
- 抽签结果一旦生成不可修改。
- 两位参与者不能抽到相同主题。
- 歌曲提交后 MVP 阶段可以允许编辑一次，降低用户输入错误成本。
- 结果页在任意阶段都可访问，但内容随状态变化。

### 14.1 用户身份识别方案

MVP 不强制登录，但需要解决"用户刷新或重新打开页面后如何找回自己的身份"的问题。

推荐方案：

- 用户加入房间后，后端返回一个临时 `sessionToken`（UUID）和对应的 `participantId`。
- 前端将 `sessionToken` 和 `participantId` 存入 `localStorage`，key 以 `roomId` 为命名空间，例如 `room_abcd12_session`。
- 后续所有 API 请求均携带 `sessionToken` 做身份校验。
- 用户刷新页面后，前端从 `localStorage` 恢复身份，无需重新加入。

防滥用：

- 同一 `roomId` 只允许注册两个 `participantId`，第三次加入直接返回房间已满。
- `sessionToken` 与 `participantId` 一一绑定，后端验证不匹配时拒绝操作。
- MVP 不需要真实账号体系，临时 Token 足够保证双人房间的基本安全。

## 16. 技术栈建议

## 19. 风险与应对

### 19.1 LLM 输出不稳定

应对：

- 使用结构化输出。
- 增加 JSON Schema 校验。
- 失败时回退默认主题池。
- 对主题长度、重复项做后处理。

### 19.2 聊天记录隐私风险

应对：

- 明确提示用户脱敏。
- MVP 不保存原始聊天记录。
- 只保存分析结果和主题。

### 19.3 网易云音乐接入不稳定

应对：

- MVP 使用歌单导出。
- 自动加入歌单作为扩展能力。
- 用户确认歌曲匹配结果后再写入歌单。

### 19.4 分享链接状态复杂

应对：

- MVP 限制两人房间。
- 使用清晰的房间状态。
- 所有关键状态以后端为准。

## 22. MVP 结论

第一版应优先做成一个小而完整的双人音乐抽签产品，而不是一开始堆复杂能力。大模型接入重点放在“聊天记录分析、主题生成、结果总结”三个高价值环节。Agent 框架在 MVP 阶段不是必须，建议先用轻量 `LLMService` 封装模型调用；当项目扩展到音乐搜索、歌单写入、多步骤状态控制时，再引入 LangGraph 会更自然，也更适合作为简历中的 Agent Workflow 升级点。
